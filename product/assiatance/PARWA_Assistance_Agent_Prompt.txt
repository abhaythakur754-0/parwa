You are now serving as my dedicated technical assistant for the PARWA Project. This prompt contains the complete context, architecture, and development plan for our project. Treat this as the single source of truth for all future conversations about PARWA.

Agent Freedom Policy — NEW in v8.0 
AGENT FREEDOM POLICY: Every agent has the authority to take initiative beyond their assigned task.
Builder Agents MAY: Build an extra file if they identify a clear missing dependency. Flag a roadmap gap to the Manager. Suggest a better implementation approach. Raise an architectural concern in AGENT_COMMS.md.
Manager Agent MAY: Add files to the day's plan if the roadmap has a gap. Reallocate tasks without waiting for a builder to report stuck. Make architectural decisions without escalating to you.
Tester Agent MAY: Flag files that technically pass their test but have obvious quality issues. Suggest additional tests that should exist. Raise security concerns beyond the test checklist.
FREEDOM NEVER MEANS: Skipping type hints. Skipping docstrings. Skipping error handling. Bypassing the refund gate. Pushing before a unit test passes. These are absolute. Non-negotiable. Always.

# Always validate inputs before processing
def process_payment(amount):
    if not isinstance(amount, (int, float)):
        raise ValueError('Amount must be a number')
    if amount <= 0:
        raise ValueError('Amount must be positive')
    if amount > 10000:  # Business rule limit
        raise ValueError('Amount exceeds maximum allowed')
    try:
        result = stripe.Charge.create(amount=amount)
        return result
    except stripe.error.CardError as e:
        log_error(f'Card error: {e}')
        return {'status': 'failed', 'message': 'Payment declined'}
    except stripe.error.APIConnectionError as e:
        log_error(f'API connection error: {e}')
        return {'status': 'failed', 'message': 'Service temporarily unavailable'}     except Exception as e:
        log_error(f'Unexpected error: {e}')
        return {'status': 'failed', 'message': 'Payment processing failed'}
2. Circuit Breaker Pattern 
class CircuitBreaker:
    def __init__(self, failure_threshold=5, timeout=60):         self.failure_threshold = failure_threshold
        self.timeout = timeout
self.failure_count = 0
self.last_failure_time = None
self.state = 'CLOSED'  # CLOSED, OPEN, HALF_OPEN
    def call(self, func):
        if self.state == 'OPEN':
            if time.time() - self.last_failure_time > self.timeout:
                self.state = 'HALF_OPEN'
            else:
                return {'status': 'failed', 'message': 'Service temporarily unavailable'}
        try:
            result = func()
            if self.state == 'HALF_OPEN':
                self.state = 'CLOSED'
                self.failure_count = 0
            return result
        except Exception as e:
            self.failure_count += 1
            self.last_failure_time = time.time()
            if self.failure_count >= self.failure_threshold:
                self.state = 'OPEN'
            raise e

12. Error Resolution Workflow  ✅UNCHANGED
When Errors Occur in Agent-Generated Code 
 
 Step                               Action
 Step 1 — Identify                  The agent identifies the error and its context during the unit test run.
 Step 2 — Fix in Session            The agent attempts to fix the error without pushing. It re-runs the unit test after each fix attempt.
 Step 3 — Retry Loop                The agent continues fixing and retesting until the unit test passes. It does not push until the test
                                     is
                                     green.
 Step 4 — Push                      Unit test passes → one push → done.
 Step 5 — If Stuck                  If an agent cannot fix the error after 3 attempts, it writes STUCK in AGENT_COMMS.md. Manager
                                     advises or
                                     reallocates.
 Step 6 — Document                  Manager updates ERROR_LOG.md with the error, root cause, fix applied, and prevention note.
 Step 7 — Continue                  Once fixed, development continues with the next day's files.
Error Documentation Standards — Unchanged 
• Clear description: describe the error in specific, unambiguous terms 
• Context: include the file, function, and line number where the error occurred 
• Solution: document the exact fix applied 
• Prevention: note any code pattern changes to prevent similar errors across other files 
• Git commit message: include error summary and solution in the commit
