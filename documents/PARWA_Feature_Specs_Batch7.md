# F-075: Batch Approval (Semantic Clusters)

## Overview

Batch Approval enables supervisors to approve entire clusters of semantically similar AI actions in a single operation, dramatically reducing review fatigue. The system groups pending approval items using embedding-based similarity (from F-071 Semantic Clustering) and only surfaces batch-approve options when the cluster's confidence score exceeds the tenant's configured threshold.

## APIs

| Endpoint | Method | Key Params | Response |
|----------|--------|------------|----------|
| `/api/approval/clusters` | GET | `?min_confidence=80&limit=20` | `[{cluster_id, ticket_count, avg_confidence, representative_ticket, intent_label}]` |
| `/api/approval/clusters/{id}/approve` | POST | `cluster_id`, `approver_user_id` | `{approved_count, rejected_count, errors[]}` |
| `/api/approval/clusters/{id}/details` | GET | `cluster_id`, `?page=1&per_page=20` | `{cluster: {...}, tickets: [{id, subject, confidence, proposed_action}]}` |

The batch approve endpoint atomically updates all `approval_records` within the cluster. Each individual ticket's `status` transitions from `pending` to `approved`, and downstream actions (email send, refund, status change) are dispatched via Celery tasks. Partial failures return the list of failed ticket IDs for retry.

## DB Tables

- **`semantic_clusters`** — `id (UUID PK)`, `company_id (UUID, indexed)`, `intent_label (VARCHAR)`, `embedding_center (VECTOR)`, `avg_confidence (DECIMAL)`, `ticket_count (INT)`, `status (VARCHAR: pending/partial/approved/rejected)`, `created_at (TIMESTAMPTZ)`, `expires_at (TIMESTAMPTZ)`
- **`cluster_tickets`** — `id (UUID PK)`, `cluster_id (UUID FK)`, `approval_record_id (UUID FK → approval_records)`, `similarity_score (DECIMAL)`, `confidence_score (DECIMAL)`
- **`approval_records`** — Updated: `cluster_id (UUID FK, nullable)` added to link individual approvals to batch operations

## BC Rules

- **BC-009 (Approval Workflow):** Every batch approval creates individual approval_records per ticket. The batch operation is atomic — either all succeed or failures are reported. Financial actions within a cluster still require supervisor+ role.
- **BC-007 (AI Model Interaction):** Clustering uses embedding vectors generated by the Smart Router (Light tier). Confidence thresholds are stored per-company, not hardcoded.
- **BC-004 (Background Jobs):** Post-approval action dispatch (email send, refund execution) runs as Celery tasks with `max_retries=3`, exponential backoff, and `company_id` as first parameter.
- **BC-001 (Multi-Tenant Isolation):** All cluster queries include `WHERE company_id = :tenant_id`. Embedding comparisons never cross tenant boundaries.

## Edge Cases

| Edge Case | Handling |
|-----------|----------|
| **Cluster contains mix of refund and non-refund actions** | System splits into sub-clusters by action_type before presenting. Supervisor can approve each sub-cluster independently. |
| **Cluster confidence below tenant threshold** | Batch approve button is disabled. Individual approval still available per ticket. Cluster displayed with warning badge. |
| **Partial failure during batch execution** | Successfully approved tickets proceed. Failed tickets remain `pending` with error details. Response includes `failed_ticket_ids` for targeted retry. |
| **Cluster expires before review** | Clusters older than 72h are dissolved back to individual pending items per F-082 timeout rules. |

## Acceptance

1. Given 15 pending refund tickets with avg cluster confidence of 92%, When the supervisor clicks "Approve Cluster", Then all 15 approval_records transition to `approved` and downstream refund actions are dispatched within 5 seconds.
2. Given a cluster with avg confidence of 65% (below the 80% threshold), When the cluster is displayed, Then the batch approve button is disabled and a warning badge shows "Low confidence — review individually."
3. Given a batch approval where 3 of 20 downstream Celery tasks fail, When the response returns, Then 17 tickets show `approved` status, 3 remain `pending` with error messages, and the response includes the 3 failed ticket IDs.
4. Given a cluster containing 5 refund actions and 10 status-change actions, When the cluster is loaded, Then it is split into two sub-clusters and each is presented with its own approve/reject controls.

# F-079: Confidence Score Breakdown

## Overview

Confidence Score Breakdown provides a visual, drill-down interface showing the individual component scores that compose each AI response's overall confidence rating. Supervisors can click into any ticket's confidence score to see retrieval relevance, intent match strength, sentiment alignment, and historical accuracy — enabling informed approve/reject decisions and identifying which model components need improvement.

## APIs

| Endpoint | Method | Key Params | Response |
|----------|--------|------------|----------|
| `/api/confidence/{ticket_id}` | GET | `ticket_id` (UUID) | `{overall: 85, components: {retrieval: 92, intent: 88, sentiment: 78, history: 82}, model_used, tier, created_at}` |
| `/api/confidence/{ticket_id}/history` | GET | `ticket_id`, `?limit=10` | `[{component, score, trend, previous_scores[]}]` |
| `/api/confidence/thresholds` | GET | — | `{action_type: {auto_approve: 90, require_review: 70, auto_reject: 40}}` (per-company config) |
| `/api/confidence/thresholds` | PUT | `{action_type, auto_approve, require_review}` | `{updated: true}` (supervisor+ only) |

The breakdown endpoint returns scores as integers (0-100) with per-component trend indicators (improving/declining/stable) based on the last 10 interactions. Threshold configuration requires supervisor+ role per BC-011 and is audit-logged.

## DB Tables

- **`confidence_scores`** — `id (UUID PK)`, `company_id (UUID, indexed)`, `ticket_id (UUID FK → tickets, unique)`, `overall_score (SMALLINT)`, `retrieval_score (SMALLINT)`, `intent_score (SMALLINT)`, `sentiment_score (SMALLINT)`, `history_score (SMALLINT)`, `model_id (VARCHAR)`, `tier (VARCHAR: light/medium/heavy)`, `created_at (TIMESTAMPTZ)`
- **`confidence_thresholds`** — `id (UUID PK)`, `company_id (UUID, unique)`, `action_type (VARCHAR: refund/status_change/email/general)`, `auto_approve_threshold (SMALLINT, DEFAULT 90)`, `require_review_threshold (SMALLINT, DEFAULT 70)`, `auto_reject_threshold (SMALLINT, DEFAULT 40)`, `updated_by (UUID FK → users)`, `updated_at (TIMESTAMPTZ)`

## BC Rules

- **BC-007 (AI Model Interaction):** Confidence thresholds are stored per-company in the database (BC-007 Rule 9), not hardcoded. The Smart Router passes each component score from the LLM evaluation pipeline. PII is redacted before any confidence-related context is sent to the LLM.
- **BC-009 (Approval Workflow):** The confidence breakdown is the primary input for approval decisions. Items below `require_review_threshold` always enter the approval queue. Items above `auto_approve_threshold` are eligible for auto-handle rules (F-077).
- **BC-005 (Real-Time Communication):** Confidence score updates are emitted via Socket.io to the approval queue dashboard in real-time. Events stored in `event_buffer` for reconnection recovery.
- **BC-001 (Multi-Tenant Isolation):** Thresholds and scores are strictly tenant-scoped. No cross-tenant confidence data is ever exposed.

## Edge Cases

| Edge Case | Handling |
|-----------|----------|
| **Component score is 0 (no data)** | Displayed as "N/A" with grey indicator. Overall score is weighted average of available components only. Does not block approval flow. |
| **Supervisor lowers auto-approve threshold to 50%** | Change is audit-logged. Previous auto-approved items are NOT retroactively affected. A warning banner notes the increased risk. |
| **Sentiment score drops 30pts between consecutive tickets** | Trend indicator shows "declining" with red arrow. The approval queue surfaces a system alert: "Sentiment confidence declining — consider manual review." |
| **All components score above 90 but overall is 85** | Tooltip explains the weighted formula. No action blocked. Transparency maintained. |

## Acceptance

1. Given a ticket with overall confidence of 72, When the supervisor clicks the confidence badge, Then a breakdown modal shows retrieval: 88, intent: 92, sentiment: 55, history: 82 with color-coded bars and trend arrows.
2. Given a supervisor with the appropriate role, When they update the auto-approve threshold from 90 to 85 for refund actions, Then the change persists and is audit-logged with the supervisor's user_id and timestamp.
3. Given a ticket where the retrieval component score is 0, When the breakdown is rendered, Then the retrieval bar shows "N/A" in grey and the overall score is calculated from the remaining three components.
4. Given 5 new tickets enter the approval queue, When their confidence scores are computed, Then Socket.io emits `approval:confidence_update` events for each to the tenant-scoped room within 2 seconds.

# F-081: Approval Reminders (2h, 4h, 8h, 24h)

## Overview

Approval Reminders implements an escalating notification ladder that alerts supervisors of pending approval items at 2-hour, 4-hour, 8-hour, and 24-hour intervals after they enter the queue. Notifications escalate in urgency across channels — starting with in-app badges, progressing to email summaries, and culminating in push notifications with count-up badges — ensuring critical items are never silently ignored.

## APIs

| Endpoint | Method | Key Params | Response |
|----------|--------|------------|----------|
| `/api/approval/reminders/preferences` | GET | — | `{channels: {in_app: true, email: true, push: true}, escalation_schedule: ["2h","4h","8h","24h"]}` |
| `/api/approval/reminders/preferences` | PUT | `{channels, escalation_schedule}` | `{updated: true}` |
| `—` | Celery Beat | `company_id`, `reminder_level` | Dispatches notifications per configured channels |

Notification dispatch is handled entirely by Celery beat — no REST endpoint triggers reminders. The beat schedule runs every 15 minutes, querying for approval_records where `status = 'pending'` and `last_reminded_at < now() - interval`. Each escalation level increases urgency: Level 1 (2h) = in-app only; Level 2 (4h) = in-app + email; Level 3 (8h) = in-app + email + push; Level 4 (24h) = all channels + cc to tenant admin.

## DB Tables

- **`approval_records`** — Updated: `last_reminded_at (TIMESTAMPTZ, nullable)`, `reminder_level (SMALLINT, DEFAULT 0)` columns added
- **`approval_reminder_log`** — `id (UUID PK)`, `company_id (UUID, indexed)`, `approval_record_id (UUID FK)`, `reminder_level (SMALLINT)`, `channel (VARCHAR: in_app/email/push)`, `recipient_user_id (UUID FK)`, `delivered_at (TIMESTAMPTZ)`, `read_at (TIMESTAMPTZ, nullable)`
- **`approval_reminder_preferences`** — `id (UUID PK)`, `company_id (UUID, unique)`, `in_app_enabled (BOOLEAN, DEFAULT true)`, `email_enabled (BOOLEAN, DEFAULT true)`, `push_enabled (BOOLEAN, DEFAULT true)`, `updated_at (TIMESTAMPTZ)`

## BC Rules

- **BC-004 (Background Jobs):** Reminder dispatch is a Celery beat task running every 15 minutes. `company_id` is the first parameter. `max_retries=3` with exponential backoff. Task logged in `task_logs`. DLQ captures permanent failures.
- **BC-006 (Email Communication):** Email reminders use Brevo template #8 (approval reminder). Rate limited to 5 emails per user per hour. OOO/auto-reply detection on replies. Logged in `email_logs`.
- **BC-009 (Approval Workflow):** Reminders do not modify approval state. They are informational only. The 72h timeout (F-082) operates independently.
- **BC-005 (Real-Time Communication):** In-app reminder badges are delivered via Socket.io emit to `tenant_{company_id}` room. Stored in `event_buffer` for reconnection.

## Edge Cases

| Edge Case | Handling |
|-----------|----------|
| **Supervisor is on vacation (OOO auto-reply)** | Email reminder logs the OOO detection. System escalates to next supervisor in the approval rotation or tenant admin at Level 3 (8h). |
| **500 pending approvals from a burst** | Email reminder batches into a single summary email (max 50 items per email, paginated). In-app shows count badge. No 500 individual emails sent. |
| **All supervisors dismissed reminder but didn't act** | Level 4 (24h) cc's the tenant admin with full pending list. Does not auto-approve or auto-reject — that is F-082's role at 72h. |
| **Celery beat misses a cycle** | Next beat run queries `WHERE last_reminded_at < now() - interval` so missed reminders are caught on the next execution. No permanent skip. |

## Acceptance

1. Given an approval_record pending for 2 hours, When the Celery beat runs, Then an in-app notification badge appears in the approval queue for all supervisors of that tenant.
2. Given an approval_record pending for 4 hours, When the escalation level triggers, Then an email is sent via Brevo template #8 listing the pending item with a direct link to the approval queue.
3. Given an approval_record pending for 24 hours, When Level 4 triggers, Then push notification, email, and in-app badge are all sent, and the tenant admin is CC'd on the email.
4. Given a supervisor has disabled email reminders in preferences, When Level 2 triggers, Then only the in-app notification is dispatched and no email is sent.

# F-082: Approval Timeout (72h Auto-Reject)

## Overview

Approval Timeout automatically rejects AI-suggested actions that remain unreviewed for 72 hours, preventing stale items from accumulating indefinitely in the approval queue. The system logs the timeout reason, notifies the original requester, and optionally re-queues the action for fresh AI re-evaluation if the underlying ticket is still open and relevant.

## APIs

| Endpoint | Method | Key Params | Response |
|----------|--------|------------|----------|
| `—` | Celery Beat | `company_id` | Scans for timed-out approvals and processes them |
| `/api/approval/timeouts` | GET | `?status=all&from_date=...` | Paginated list of timeout events with reason and re-queue status |
| `/api/approval/{id}/reinstate` | POST | `approval_record_id` | `{reinstated: true, new_status: "pending"}` (supervisor+ only) |

The core logic runs as a Celery beat task every 30 minutes. It queries `approval_records WHERE status = 'pending' AND created_at < now() - interval '72 hours'`. Each timed-out record is set to `status = 'timeout_rejected'`, and the related ticket is flagged for re-evaluation.

## DB Tables

- **`approval_records`** — Updated: `status` enum extended with `timeout_rejected`, `timeout_rejected_at (TIMESTAMPTZ, nullable)`, `timeout_reason (TEXT, nullable)`, `reinstate_count (SMALLINT, DEFAULT 0)`
- **`approval_timeout_log`** — `id (UUID PK)`, `company_id (UUID, indexed)`, `approval_record_id (UUID FK)`, `ticket_id (UUID FK)`, `original_action (VARCHAR)`, `timeout_at (TIMESTAMPTZ)`, `ticket_still_open (BOOLEAN)`, `re_queued (BOOLEAN)`, `reinstated_by (UUID FK, nullable)`, `reinstated_at (TIMESTAMPTZ, nullable)`

## BC Rules

- **BC-004 (Background Jobs):** Timeout processing is a Celery beat task with `max_retries=3`. `company_id` is the first parameter. Task logged in `task_logs`. Re-queue actions dispatched as separate Celery tasks.
- **BC-009 (Approval Workflow):** Timeout rejection is distinct from manual rejection. Timed-out items can be reinstated by a supervisor, incrementing `reinstate_count`. Reinstated items get a fresh 72-hour window. Items with `reinstate_count >= 3` cannot be reinstated and require a new AI evaluation.
- **BC-002 (Financial Actions):** Financial actions (refunds, credits) that timeout are NOT auto-executed. They are rejected and require a fresh approval. The timeout is logged in `audit_trail`.
- **BC-012 (Error Handling & Resilience):** If the timeout task itself fails, DLQ captures it. On recovery, the next beat run processes any missed timeouts based on timestamp comparison.

## Edge Cases

| Edge Case | Handling |
|-----------|----------|
| **Ticket resolved by human while approval was pending** | Timeout still fires but sets `ticket_still_open = false`. No re-queue. Log entry notes "Ticket already resolved." |
| **Reinstated item times out again** | `reinstate_count` increments. After 3 reinstatements, the item is permanently closed with reason "max reinstatements exceeded." |
| **Bulk timeout (50 items expire in same beat cycle)** | Processed in batches of 10 per Celery task to avoid long-running tasks. Each batch is a separate Celery task with its own retry chain. |
| **Timeout fires during system maintenance** | Beat task resumes on next cycle. No approvals are missed because the `created_at < now() - 72h` query is idempotent. |

## Acceptance

1. Given an approval_record created 73 hours ago with `status = 'pending'`, When the timeout beat task runs, Then the status transitions to `timeout_rejected` and the timeout log is created.
2. Given a timed-out financial action (refund of $45.00), When the timeout processes, Then the refund is NOT executed, and the `audit_trail` logs `action: "approval_timeout_reject"` with the amount.
3. Given a supervisor reinstates a timed-out approval, When the reinstatement completes, Then the approval status returns to `pending` with a fresh 72-hour window and `reinstate_count` increments.
4. Given an approval with `reinstate_count = 3` that times out again, When the timeout processes, Then the item is permanently closed and cannot be reinstated via the API.

# F-084: Undo System (Action + Email Recall)

## Overview

The Undo System provides a safety net for reversing AI-executed actions and recalling sent emails within a configurable time window. When a supervisor discovers an erroneous AI action (incorrect refund, wrong status change, inappropriate email), they can trigger an undo that atomically reverses the primary action, compensates any side effects, and if applicable, recalls the sent email via the Brevo API — all within a defined recall window.

## APIs

| Endpoint | Method | Key Params | Response |
|----------|--------|------------|----------|
| `/api/actions/{id}/undo` | POST | `action_id` (UUID), `reason (TEXT)` | `{undone: true, reversal_details: {...}, email_recalled: bool}` |
| `/api/actions/{id}/undo/check` | GET | `action_id` (UUID) | `{undoable: bool, reason_if_not, window_remaining_seconds}` |
| `/api/actions/undo/history` | GET | `?ticket_id=...&limit=20` | Paginated undo history with reversal status |

The undo endpoint validates the recall window, checks for dependent actions, and executes a compensation transaction. For email recall, it calls the Brevo API to recall the message. All financial reversals (refund reversals) follow BC-002 atomic transaction rules.

## DB Tables

- **`executed_actions`** — `id (UUID PK)`, `company_id (UUID, indexed)`, `ticket_id (UUID FK)`, `action_type (VARCHAR: refund/status_change/email/tag_assignment)`, `action_payload (JSONB)`, `executed_by (VARCHAR: ai/auto/manual)`, `executed_at (TIMESTAMPTZ)`, `undoable_until (TIMESTAMPTZ)`, `undone (BOOLEAN, DEFAULT false)`, `undo_reason (TEXT, nullable)`, `undone_by (UUID FK → users, nullable)`, `undone_at (TIMESTAMPTZ, nullable)`
- **`undo_log`** — `id (UUID PK)`, `company_id (UUID, indexed)`, `executed_action_id (UUID FK)`, `reversal_actions (JSONB)`, `email_recalled (BOOLEAN)`, `financial_reversal_amount (DECIMAL(10,2), nullable)`, `undo_duration_ms (INTEGER)`, `status (VARCHAR: success/partial_failure/failed)`, `error_details (TEXT, nullable)`

## BC Rules

- **BC-002 (Financial Actions):** Refund reversals are financial operations. They require atomic DB transactions, are logged in `audit_trail`, use `DECIMAL(10,2)` for amounts, and include idempotency checks. Paddle API calls for refund reversal are idempotent.
- **BC-006 (Email Communication):** Email recall uses the Brevo API. Recall attempts are logged in `email_logs`. If Brevo recall fails (email already opened), the undo still reverses other side effects but flags `email_recalled = false`.
- **BC-009 (Approval Workflow):** Undo is only available for AI-executed or auto-approved actions. Manually approved actions cannot be undone — the human approver is the authority. Undone items are logged in the approval audit trail.
- **BC-012 (Error Handling & Resilience):** Partial undo (e.g., refund reversed but email recall failed) is logged as `partial_failure` with `error_details`. The system never leaves silent partial state.

## Edge Cases

| Edge Case | Handling |
|-----------|----------|
| **Email already opened by customer** | Brevo recall returns failure. Undo reverses other side effects (refund, status) but `email_recalled = false`. Alert sent to supervisor with follow-up template suggestion. |
| **Action outside recall window** | `/undo/check` returns `{undoable: false, reason: "recall_window_expired"}`. The undo endpoint returns HTTP 403 with a clear message. |
| **Dependent actions exist (e.g., status change triggered a notification)** | Undo reverses all actions in the dependency chain atomically. If any reversal fails, the entire transaction rolls back. |
| **Undo a refund that was already partially processed by payment provider** | System calls Paddle refund reversal API. If Paddle returns "already_settled," the financial reversal is flagged and the amount is recorded as a receivable credit in `audit_trail`. |

## Acceptance

1. Given an AI-executed refund of $50.00 within the recall window, When the supervisor clicks "Undo", Then the refund is reversed via Paddle, the ticket status reverts, and the `audit_trail` logs the reversal with the original and reversed amounts.
2. Given an AI-sent email that was not yet opened, When the undo is triggered within 10 minutes, Then the Brevo recall API is called and `email_recalled = true` in the undo log.
3. Given an action that was manually approved by a human supervisor, When the undo check is called, Then `undoable = false` with reason "manually_approved_actions_cannot_be_undone."
4. Given an undo where the refund reversal succeeds but the email recall fails, When the undo completes, Then the status is `partial_failure`, the refund is reversed, and the supervisor receives an alert about the email recall failure.

# F-088: System Status Panel

## Overview

The System Status Panel provides a real-time health dashboard showing the operational state of every PARWA subsystem — LLM provider availability, Celery queue depths, approval backlog count, active agent statuses, integration connectivity, and Redis/database connectivity. It gives operators a single pane of glass to detect degradation before it impacts customers.

## APIs

| Endpoint | Method | Key Params | Response |
|----------|--------|------------|----------|
| `/api/system/status` | GET | — | `{subsystems: {llm: {status, latency_ms, active_providers}, celery: {queue_depth, worker_count, dlq_size}, redis: {connected, memory_usage}, postgres: {connected, connection_pool}, integrations: [{name, status, last_ping}], agents: {active, training, paused, total}}}` |
| `/api/system/status/history` | GET | `?hours=24` | Time-series snapshots for charting (1 per minute) |
| `/api/system/status/incidents` | GET | `?active_only=true` | Current active incidents with duration |

All status data is aggregated from Redis health keys (updated every 10 seconds by background tasks) plus direct connectivity probes. The endpoint responds within 200ms and is cached in Redis with a 5-second TTL.

## DB Tables

- **`system_health_snapshots`** — `id (UUID PK)`, `company_id (UUID, nullable — global for platform ops)`, `subsystem (VARCHAR)`, `status (VARCHAR: healthy/degraded/down)`, `latency_ms (INTEGER, nullable)`, `metadata (JSONB)`, `recorded_at (TIMESTAMPTZ)`
- **`system_incidents`** — `id (UUID PK)`, `subsystem (VARCHAR)`, `severity (VARCHAR: low/medium/high/critical)`, `status (VARCHAR: active/resolved)`, `description (TEXT)`, `detected_at (TIMESTAMPTZ)`, `resolved_at (TIMESTAMPTZ, nullable)`, `resolution_notes (TEXT, nullable)`

Redis keys: `parwa:health:{subsystem}` (TTL 30s), `parwa:health:global` (aggregated, TTL 5s)

## BC Rules

- **BC-005 (Real-Time Communication):** Status updates pushed via Socket.io to `tenant_{company_id}` rooms every 10 seconds. Events buffered in `event_buffer`. Graceful degradation: full status available via REST GET even if Socket.io is down.
- **BC-012 (Error Handling & Resilience):** If a subsystem probe fails, the panel shows `degraded` (not `down`) until 3 consecutive probe failures confirm `down`. Prevents flickering status.
- **BC-001 (Multi-Tenant Isolation):** Global subsystem health (LLM, Redis, Postgres) is shared across tenants. Tenant-specific data (agent count, queue depth) is scoped by `company_id`.
- **BC-004 (Background Jobs):** Health probes run as Celery beat tasks every 10 seconds. `company_id` passed as first parameter. Probes are lightweight (TCP connect + latency measurement).

## Edge Cases

| Edge Case | Handling |
|-----------|----------|
| **Redis itself is down** | Status panel falls back to direct connectivity probes for Postgres and external APIs. Shows Redis as `down`. Slower response time (500ms vs 200ms). |
| **All LLM providers show degraded** | Panel flashes critical severity. Jarvis auto-opens with suggestion: "All LLM providers degraded — consider enabling Trust Preservation Protocol (F-094)." |
| **Socket.io server unreachable** | Panel still loads via REST. Status badge shows "Real-time updates unavailable — refreshing manually." |
| **Probe task itself is in DLQ** | Health snapshot ages. Panel detects stale data (> 60s old) and shows "Health data may be stale" warning badge. |

## Acceptance

1. Given all subsystems are operational, When the System Status Panel loads, Then every subsystem shows a green "healthy" badge and the aggregate status is "All Systems Operational."
2. Given the Brevo API returns 500 for 3 consecutive probes, When the status updates, Then the Email subsystem shows "down" in red and a system_incident is created with severity "high."
3. Given a Socket.io disconnection, When the panel is visible, Then it continues to show the last known status and displays a "Real-time updates unavailable" banner, polling via REST every 30 seconds.
4. Given the Celery default queue has 500 pending tasks, When the status loads, Then the Celery subsystem shows "degraded" with the queue depth of 500 and worker count.

# F-089: GSD State Terminal

## Overview

The GSD State Terminal is a live debugging interface that shows the real-time execution state of the GSD (Guided Sequential Dialog) state machine for any active ticket. It displays the current step, next step, pending decisions, state variables, and transition history — enabling operators and engineers to diagnose stuck conversations, unexpected loops, and state machine errors without digging through logs.

## APIs

| Endpoint | Method | Key Params | Response |
|----------|--------|------------|----------|
| `/api/gsd/state/{ticket_id}` | GET | `ticket_id` (UUID) | `{current_step, next_step, state_variables: {...}, transition_history: [{from, to, timestamp, trigger}], pending_decisions: [{type, options, timeout_at}], started_at, last_activity_at}` |
| `/api/gsd/state/{ticket_id}/force-transition` | POST | `ticket_id`, `target_step`, `reason` | `{success, new_state}` (admin only) |
| `/api/gsd/sessions/active` | GET | `?agent_id=...&stuck_only=true` | List of active GSD sessions with current step and duration |

- **BC-008 (State Management):** GSD state is stored in Redis as primary and PostgreSQL as fallback (BC-008 Rule 8). Every state update writes to both. On Redis restart, states are recovered from PostgreSQL. The terminal reads from Redis for real-time and falls back to PostgreSQL.
- **BC-005 (Real-Time Communication):** State transitions are emitted via Socket.io to `tenant_{company_id}` room in real-time. The terminal subscribes to `gsd:transition:{ticket_id}` events. Buffered in `event_buffer`.
- **BC-007 (AI Model Interaction):** GSD steps that involve LLM calls route through the Smart Router. If the LLM call is the pending decision, the terminal shows the LLM request context (redacted) and timeout countdown.

| Edge Case | Handling |

| **Force-transition to an invalid step** | API returns HTTP 400 with valid step options. No state change made. Error logged. |

1. Given an active ticket in step "awaiting_customer_response", When the GSD State Terminal is opened, Then it displays the current step, next step ("classify_intent"), state variables, and a full transition history with timestamps.
2. Given a GSD session stuck on step "llm_processing" for 35 minutes, When the terminal renders, Then a red "STUCK" badge appears and the force-transition button is enabled with valid target step options.
3. Given an admin force-transitions a session from "awaiting_refund_approval" to "refund_completed", When the transition executes, Then the new state is written to both Redis and PostgreSQL, a transition record is created with `trigger = 'force'`, and the action is audit-logged.

## Overview

Train from Error is a one-click action available on any error entry that packages the error context, the associated ticket data, the AI's failed response, and the correct expected outcome into a structured training data point. The train endpoint extracts the error context, pairs it with the ticket's ground truth (if available), and creates a `training_data_points` record. ## Edge Cases

3. Given an error with no associated ticket, When "Train from Error" is clicked, Then a training point is created with `ticket_id = null` and the input context is derived from the error log entry.
4. Given the same error is clicked "Train from Error" twice, When the second request is processed, Then no duplicate is created and the API returns the existing training_point_id.

# F-094: Trust Preservation Protocol

## Overview

The Trust Preservation Protocol ensures PARWA never tells a customer "we can't help you" when external systems fail. Instead of surfacing raw errors, the system queues the intended action, delivers a reassuring holding message ("We're looking into this for you right away"), and processes the action in the background once the external system recovers — preserving customer trust even during infrastructure degradation.

## APIs

| Endpoint | Method | Key Params | Response |
|----------|--------|------------|----------|
| `/api/trust/queue` | POST (internal) | `ticket_id`, `intended_action (JSONB)`, `fallback_message (TEXT)` | `{queued: true, estimated_retry_at}` |
| `/api/trust/status/{ticket_id}` | GET | `ticket_id` (UUID) | `{status: processing/queued/delivered, fallback_sent: bool, retry_count}` |
| `/api/trust/queue` | GET | `?status=queued&limit=20` | Paginated list of trust-preserved actions awaiting retry |

This feature is primarily an internal protocol, not a user-facing API. External services call `/api/trust/queue` when they detect an external dependency failure. The protocol then manages the retry lifecycle.

## DB Tables

- **`trust_preservation_queue`** — `id (UUID PK)`, `company_id (UUID, indexed)`, `ticket_id (UUID FK → tickets)`, `intended_action (JSONB)`, `fallback_message (TEXT)`, `retry_count (SMALLINT, DEFAULT 0)`, `max_retries (SMALLINT, DEFAULT 10)`, `status (VARCHAR: queued/retrying/delivered/failed/escalated)`, `original_error (TEXT)`, `next_retry_at (TIMESTAMPTZ)`, `delivered_at (TIMESTAMPTZ, nullable)`, `escalated_at (TIMESTAMPTZ, nullable)`, `created_at (TIMESTAMPTZ)`

## BC Rules

- **BC-012 (Error Handling & Resilience):** Core building code. This protocol IS the error handling strategy for customer-facing interactions. No raw errors reach customers. All failures are wrapped in reassuring messaging.
- **BC-007 (AI Model Interaction):** Fallback messages are generated by the Smart Router (Light tier) with Guardrails evaluation. Messages are cached per intent type to avoid unnecessary LLM calls during outages.
- **BC-005 (Real-Time Communication):** Status updates emitted via Socket.io. The approval queue shows trust-preserved items with a "held" badge. Operators can monitor the retry queue in real-time.
- **BC-004 (Background Jobs):** Retry processing runs as a Celery beat task every 60 seconds. `company_id` is the first parameter. `max_retries=3` per attempt, `max_retries=10` total for the trust queue item.

## Edge Cases

| Edge Case | Handling |
|-----------|----------|
| **Customer follows up while action is queued** | System responds with "Your request is still being processed. We'll update you shortly." Does not create a duplicate queue entry. |
| **All retry attempts exhausted (10 retries over ~30 min)** | Item escalated to human agent. Customer receives "A team member is now personally handling your request." |
| **External system recovers mid-queue** | Beat task detects recovery via probe. Accelerates retry schedule — processes all queued items for that subsystem immediately. |
| **Trust protocol itself fails (Celery down)** | Items remain in PostgreSQL queue. On Celery recovery, the beat task processes them. Redis AOF preserves task metadata. |

## Acceptance

1. Given the Brevo API is down and an AI response needs to email a customer, When Trust Preservation activates, Then the customer receives "We're looking into this for you right away" and the email is queued for background retry.
2. Given 5 queued actions for the same external system, When the system recovers, Then all 5 actions are processed within 2 minutes of recovery detection and the trust queue items transition to `delivered`.
3. Given a trust queue item has exhausted 10 retries, When the max is reached, Then the item is escalated to a human agent and the customer receives a personalized handoff message.
4. Given the Smart Router is completely unavailable, When a fallback message is needed, Then the system uses a pre-cached fallback template (no LLM call required) and the message still passes Guardrails evaluation.

## Overview

Add/Scale Agent is the provisioning flow that triggers a Paddle checkout when a tenant wants to add a new agent beyond their plan's inclusion. After successful payment, the system auto-provisions the agent with base configuration, assigns permissions based on the plan tier, grants integration access, and initiates the Agent Lightning Training Loop (F-100) — creating a seamless purchase-to-production pipeline.

## APIs

| Endpoint | Method | Key Params | Response |
|----------|--------|------------|----------|
| `/api/agents/provision/checkout` | POST | `{agent_name, specialty, channels}` | `{paddle_checkout_url, pending_agent_id}` |
| `/api/webhooks/paddle/agent-provisioned` | POST (webhook) | Paddle webhook payload | `{status: "provisioned"}` (internal processing) |
| `/api/agents/provision/{id}/status` | GET | `pending_agent_id` (UUID) | `{status: awaiting_payment/provisioning/training/active, payment_status}` |

The checkout endpoint creates a `pending_agents` record and generates a Paddle checkout URL with custom attributes (company_id, agent_name, specialty). On Paddle webhook confirmation (subscription or one-time charge), the provisioning pipeline executes.

## DB Tables

- **`pending_agents`** — `id (UUID PK)`, `company_id (UUID, indexed)`, `agent_name (VARCHAR)`, `specialty (VARCHAR)`, `channels (JSONB)`, `paddle_checkout_id (VARCHAR)`, `paddle_transaction_id (VARCHAR, nullable)`, `payment_status (VARCHAR: pending/paid/failed/refunded)`, `provisioning_status (VARCHAR: awaiting_payment/provisioning/training/active/failed)`, `created_at (TIMESTAMPTZ)`, `provisioned_at (TIMESTAMPTZ, nullable)`
- **`agents`** — See F-095. Created during provisioning with `status = 'training'`.
- **`webhook_events`** — See BC-003. Paddle webhook events stored for idempotency.

## BC Rules

- **BC-002 (Financial Actions):** Agent provisioning is a financial operation. Paddle checkout uses idempotency keys. Webhook processing follows BC-003 rules. All amounts stored as `DECIMAL(10,2)`. Audit trail logged in `audit_trail`. If Paddle call fails after DB update, compensation transaction rolls back.
- **BC-003 (Webhook Handling):** Paddle webhook for agent provisioning is verified via HMAC-SHA256. Idempotency check on `paddle_event_id`. Async processing via Celery. Response within 3 seconds.
- **BC-001 (Multi-Tenant Isolation):** Pending agents and provisioned agents are tenant-scoped. Paddle checkout custom attributes include `company_id` for webhook correlation.
- **BC-011 (Authentication & Security):** Checkout initiation requires supervisor+ role. Paddle webhook verification prevents spoofed provisioning requests.

## Edge Cases

| Edge Case | Handling |
|-----------|----------|
| **Paddle payment fails after checkout initiated** | `pending_agents.payment_status` set to `failed`. User can retry from the dashboard. No agent record created. |
| **Paddle webhook arrives but provisioning fails** | Webhook stored with `status = processed`. Provisioning retry via Celery with `max_retries=3`. If all retries fail, `provisioning_status = 'failed'` and ops team alerted. |
| **User clicks "Add Agent" but changes mind before paying** | `pending_agents` record remains with `payment_status = pending`. A Celery beat task cleans up stale pending records after 24 hours. |
| **Duplicate webhook (Paddle sends twice)** | Idempotency check on `paddle_event_id` returns 200 without re-provisioning. BC-003 Rule 6. |

## Acceptance

1. Given a supervisor clicks "Add Agent" and fills in the form, When the checkout endpoint is called, Then a Paddle checkout URL is returned and a `pending_agents` record is created with `payment_status = 'pending'`.
2. Given Paddle sends a successful payment webhook, When the webhook is processed, Then the agent is provisioned with base configuration, status set to `training`, and F-100 (Lightning Training Loop) is triggered.
3. Given Paddle sends the same webhook twice, When the second webhook arrives, Then the idempotency check finds the existing event and returns 200 without re-provisioning.
4. Given provisioning fails after successful payment, When all 3 Celery retries are exhausted, Then the ops team is alerted via Slack and email, and the `provisioning_status` is set to `failed` with a manual retry option.

# F-107: Cold Start Problem Handler

## Overview

The Cold Start Problem Handler provides specialized onboarding for newly created agents that have zero historical data. Instead of failing on low confidence, these agents operate with general industry knowledge, pre-built response templates, a curated seed dataset, and elevated approval thresholds — ensuring they can handle basic queries from day one while progressively learning tenant-specific patterns through the Lightning Training Loop (F-100).

## APIs

| Endpoint | Method | Key Params | Response |
|----------|--------|------------|----------|
| `/api/agents/{id}/cold-start/config` | GET | `agent_id` (UUID) | `{industry, seed_dataset_size, approval_threshold_multiplier, general_kb_enabled, estimated_training_completion}` |
| `/api/agents/{id}/cold-start/config` | PUT | `{industry, custom_templates[], approval_threshold_multiplier}` | `{updated: true}` (supervisor+ only) |
| `/api/agents/{id}/cold-start/progress` | GET | `agent_id` (UUID) | `{tickets_handled, confidence_trend, auto_approve_rate, estimated_ready_date, current_mode: cold_start/transitioning/warm}` |

The cold-start mode automatically activates when an agent is first provisioned (F-099) and deactivates when the agent has handled 200+ tickets with avg confidence above the tenant's standard threshold for 7 consecutive days.

## DB Tables

- **`agent_cold_start_config`** — `id (UUID PK)`, `agent_id (UUID FK, unique)`, `company_id (UUID, indexed)`, `industry (VARCHAR)`, `approval_threshold_multiplier (DECIMAL, DEFAULT 1.5)`, `general_kb_enabled (BOOLEAN, DEFAULT true)`, `seed_dataset_applied (BOOLEAN, DEFAULT false)`, `custom_templates (JSONB, nullable)`, `status (VARCHAR: cold_start/transitioning/warm/completed)`, `tickets_handled_at_activation (INT, DEFAULT 0)`, `activated_at (TIMESTAMPTZ)`, `completed_at (TIMESTAMPTZ, nullable)`
- **`agent_cold_start_metrics`** — `id (UUID PK)`, `agent_id (UUID FK, indexed)`, `date (DATE)`, `tickets_handled (INT)`, `avg_confidence (DECIMAL)`, `auto_approve_rate (DECIMAL)`, `escalation_rate (DECIMAL)`, `mode (VARCHAR)`

## BC Rules

- **BC-007 (AI Model Interaction):** Cold-start agents use the Smart Router with elevated thresholds — the approval threshold is multiplied by 1.5x (configurable), sending more items to human review during the cold-start period. Confidence thresholds are per-company per BC-007 Rule 9.
- **BC-009 (Approval Workflow):** During cold-start, ALL financial actions require human approval regardless of confidence score. Auto-approve rules (F-077) are disabled for cold-start agents until they reach "warm" status.
- **BC-001 (Multi-Tenant Isolation):** Cold-start configurations and metrics are tenant-scoped. Seed datasets are sourced from industry templates scoped to the tenant's vertical.
- **BC-004 (Background Jobs):** Cold-start mode evaluation runs as a Celery beat task daily at midnight UTC. It checks if the agent has met the warm-up criteria (200+ tickets, 7-day confidence above threshold).

## Edge Cases

| Edge Case | Handling |
|-----------|----------|
| **Agent receives VIP ticket during cold-start** | VIP tickets (F-080) are always escalated to humans regardless of agent mode. Cold-start handling does not apply to VIP tickets. |
| **Tenant selects wrong industry during setup** | Cold-start uses general knowledge as fallback. Operator can update the industry config at any time, triggering a seed dataset refresh. |
| **Agent reaches 199 tickets but confidence is still low** | Cold-start continues. The system does not force a transition. The agent stays in cold-start until BOTH criteria are met: volume AND confidence. |
| **Cold-start agent makes 10 mistakes in first 50 tickets** | Normal mistake tracking applies (F-101). Cold-start agents have a higher tolerance (100 mistakes before forced retraining vs. 50 for warm agents). |

## Acceptance

1. Given a newly provisioned agent with 0 tickets handled, When the first ticket arrives, Then the agent operates in cold-start mode with 1.5x approval thresholds and general knowledge base responses.
2. Given a cold-start agent handles 210 tickets over 10 days with avg confidence of 82% (above the 75% standard threshold) for 7 consecutive days, When the daily evaluation runs, Then the agent transitions from "cold_start" to "warm" and standard approval thresholds apply.
3. Given a cold-start agent receives a refund request with confidence 88%, When the approval threshold is 80% * 1.5 = 86.5%, Then the item enters the approval queue despite being above the standard threshold.
4. Given a supervisor updates the cold-start industry from "general" to "ecommerce", When the config is saved, Then a new industry-specific seed dataset is applied and the agent's cold-start progress is reset.
